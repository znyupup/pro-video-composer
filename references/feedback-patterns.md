# 用户反馈应对模式(NYX 风格沉淀)

跟用户协作做视频时的高频反馈模式 + 应对策略。

## 1. "换个思路" 90% = crop 偏移参数错

**症状**:用户说"PiP 圆框无人脸,你这个地方太容易出问题,怎么改都容易出问题,换个思路搞一下"

**误判**:急着提 N 个架构思路(静态头像 / 反向重构 / 全屏数字人当底层 ...)

**正解**:先 vision 测人物在画面里的精确 x_center,改 crop 起点为 `x_center − 210`。
80% 情况这就修好了,根本不需要换架构。

**规则**:任何 PiP 视觉问题,**先 vision 看一眼人脸实际位置,再决定是不是真的需要换思路**。

## 2. 模糊反馈"白色的字" = 具体某一个,不是所有

**症状**:用户说"底图上有白色的字,你重新画一下"

**误判**:把所有白色文字 + chat 气泡 + 副标 一锅端删掉

**正解**:用户指的是**他视觉上看到的某一个具体元素**(通常是占位符/底图字),不是同类全删。

**操作前先 vision 截图给用户圈定**:"你说的是这个 [圈红] 还是包括这些 [圈红] 也要删?"
宁可多问一句,别批量误删。

## 3. 偏好"对比再选" 而不是"我推荐 X"

**用户语境**:架构性决策(crop 偏移、切换公式、视觉布局、节奏)

**反例**:"这个有 4 种方案,我推荐 B,你觉得呢?"
**正例**:"做了 A 和 B 两个版本,你看哪个顺眼"(同时跑两个推送对比)

**应用**:涉及"效果时间点"、"crop 尺寸"、"位置"、"颜色"、"节奏" 等主观决策时,做 2-3 个候选并行跑出,推用户选。

## 4. 反馈是渐进迭代,不是终态

**症状**:用户先说"删 X",改完后又说"X 应该保留"

**根因**:用户偏好不是稳定终态,是"看了再说"。每次反馈基于当前版本,不预演下一步。

**规则**:删元素时**减一半**(透明度 / 时长 / 频率)比"全删"稳。留回退空间,下次用户说"再加回点"时不用从零做。

## 5. 拼接素材的"完整版"必须问拼接点

**症状**:用户给"完整版" video 说"用这个,别自己拼了",但 video 内部其实是他自己拼的

**应对**:
1. 直接问"拼接点在哪?"
2. 用户不记得 → 用 `scripts/check-splice-points.sh` 检测
3. fade/overlay 留 1s+ 余量(0.39s 不够,详见 `splice-safety-rules.md`)

## 6. 长响应周期(60min+)

**症状**:用户响应周期波动大(30min - 1h+ 无回复)

**应对**:
- 60min 无回复 → 主动 close 当前阶段 + 归档进度 + 标 todo
- 不要发 follow-up ping
- 下次会话先读归档再接续

## 7. 指令优先级:具体 > 抽象

用户可能同时表达:
- 抽象目标:"专业一些"、"好看一些"
- 具体指令:"气泡靠右"、"延长到 3s"

**优先级**:具体指令优先做(可量化、可验证),抽象目标作为约束(不能违反)。

## 8. 数字人不能"石化"

**用户原话**:"小框的数字人也要说话,不能一直保持不动"

**含义**:数字人 PiP 必须是动态 video 不是静态 PNG。
即使切换公式更复杂、即使有拼接卡顿,也要保持"主持人正在讲"的视觉。

**规则**:除非用户明确要求,**默认 PiP = 动态 video**,不要为了规避技术问题降级成静态。

## 9. 信息密度饱和判断

**症状**:cut 1.9s 同时有 3-5 个文字元素 → 用户说"白字干扰"

**判断**:1 cut 信息饱和度 ≈ 单条核心信息 + 1 视觉装饰(图标/颜色高亮)+ 1 数字人 PiP。
超过会拥挤,需要拆 cut 或减元素。

## 10. cumsum 必须实测

**症状**:agent 凭记忆算"c01=2s c02=1s ..." 推 cut5 全屏 timing,但实际 c01 是 2.5s → timing 全错

**规则**:任何"切入时机"决策前必须 `ffprobe -show_entries format=duration` 实测每段时长,**绝不凭记忆**。
